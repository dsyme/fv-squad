# Aeneas Integration Setup

This document describes how to set up
[Aeneas](https://github.com/AeneasVerif/aeneas) — a verified Rust→Lean translation
tool — to generate faithful Lean 4 implementation models from raft-rs source code.

See `lean/FVSquad/Aeneas/` for the resulting refinement theorems.

---

## What Aeneas Does

Aeneas translates Rust MIR → a pure lambda calculus (LLBC) → Lean 4 (or F*, Coq,
HOL4).  The generated Lean code faithfully models:

* Machine integers (`Std.Usize`, `Std.U64`, etc.) with checked arithmetic in `Result`
* `Vec`, `Slice`, `Array`, `String` with full stdlib models
* Mutable references via *backward functions* (continuations for the borrow return)

This allows proving that the **actual Rust code** satisfies the abstract FVSquad
specifications, closing the gap between "the algorithm is correct" and "this
implementation is correct".

---

## Prerequisites

* Rust toolchain (stable, matching `rust-toolchain.toml` in the repo root)
* Lean 4 + Mathlib (already set up if you're working with the FVSquad proofs)
* `opam` (for Charon's OCaml dependencies) or a pre-built Charon binary

---

## Step 1: Install Charon (Rust → LLBC frontend)

Charon is the Rust compiler plugin that extracts an LLBC (Low-Level Borrow Calculus)
representation from Rust MIR.

```bash
# Clone Aeneas (which pins its Charon version)
git clone https://github.com/AeneasVerif/aeneas
cd aeneas

# Build Charon (requires opam + OCaml ≥ 4.14)
make setup-charon   # installs Charon's pinned version
make                # builds the aeneas binary
```

The resulting binary is `bin/aeneas`.

---

## Step 2: Run Charon on raft-rs

From the root of this repository:

```bash
# Build raft-rs with the `aeneas` feature (removes unsafe blocks)
cd /path/to/fv-squad
cargo build --features aeneas

# Extract LLBC — Charon hooks into cargo's build process
charon cargo --preset=aeneas --features aeneas
```

This produces `raft.llbc` (or similar) in the current directory.

> **Why `--features aeneas`?**
> The `aeneas` Cargo feature replaces the `MaybeUninit` + `unsafe` stack-array
> optimisation in `committed_index` with a semantically equivalent safe `Vec`
> implementation.  Aeneas only handles safe Rust, so this feature is required.
> See `src/quorum/majority.rs` for the `#[cfg(feature = "aeneas")]` guards.

---

## Step 3: Run Aeneas (LLBC → Lean)

```bash
# From the aeneas repo directory:
./bin/aeneas -backend lean -split-files /path/to/raft.llbc -dest /path/to/output
```

Useful flags:
* `-split-files` — one Lean file per Rust module (recommended)
* `-dest <dir>` — output directory for generated `.lean` files
* `-no-code-duplication` — avoid duplicating definitions shared across modules

---

## Step 4: Examine the Generated Output

The generated files will contain Lean 4 definitions like:

```lean
-- Auto-generated by Aeneas from src/util.rs
-- Function: pub fn majority(total: usize) -> usize
def raft.majority (total : Std.Usize) : Result Std.Usize :=
  let i := total / 2#usize
  let i1 ← i + 1#usize   -- checked addition; returns Err on overflow
  ok i1
```

The `Result` monad wraps checked arithmetic.  For `majority`, overflow can only occur
if `total / 2 + 1 > usize::MAX`, which is impossible in practice.

---

## Step 5: Write Refinement Theorems

Place generated files under `formal-verification/lean/FVSquad/Aeneas/` and write
refinement theorems connecting them to the existing FVSquad abstract specifications.

See `lean/FVSquad/Aeneas/UtilRefinements.lean` for examples and skeletons.

The general pattern:

```lean
/-- The Aeneas-generated `majority` refines the FVSquad abstract model. -/
theorem aeneas_majority_refines (n : Std.Usize) (h : n.val / 2 + 1 ≤ Std.Usize.max) :
    raft.majority n = ok ⟨FVSquad.MajorityQuorum.majority n.val, h⟩ := by
  simp [raft.majority, FVSquad.MajorityQuorum.majority]
  omega
```

## Module Feasibility Summary

| Module | Aeneas feasible today? | Notes |
|--------|----------------------|-------|
| `src/util.rs` | ✅ Yes | Pure arithmetic, no unsafe after `--features aeneas` |
| `src/log_unstable.rs` | ✅ Yes | Only `Vec`, `Option`; stub `slog::Logger` |
| `src/tracker/inflights.rs` | ✅ Yes | `Vec` only |
| `src/tracker/progress.rs` | ✅ Yes | No collections; manageable trait bounds |
| `src/quorum/majority.rs` | ✅ Yes | Requires `--features aeneas` for `committed_index` |
| `src/raft_log.rs` | ⚠️ Partial | `Vec` OK; protobuf `Entry` needs external model stub |
| `src/quorum/majority.rs` (HashSet path) | ⚠️ Blocked | No Aeneas stdlib model for `HashSet` |
| `src/tracker.rs` | ⚠️ Blocked | `HashMap<u64, Progress>` not in Aeneas stdlib |
| `src/read_only.rs` | ⚠️ Blocked | `HashMap<Vec<u8>, ReadIndexStatus>` not in Aeneas stdlib |
